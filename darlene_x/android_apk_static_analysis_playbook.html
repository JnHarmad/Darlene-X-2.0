Android APK Static Analysis Playbook — end-to-end methodology including novel semantic analysis not covered by MobSF or VirusTotal

Phase 1APK Unpacking & Reconnaissance

Before any analysis, fully decompose the APK. An APK is a ZIP. Treat every layer — classes.dex, resources, libs, META-INF — as a potential surface.

1Verify integrity & get metadata
aapt2apksignerfile
▾

Before touching content, verify the APK signature, check for multi-dex, and fingerprint the build.

# Fingerprint sha256sum app.apk file app.apk # Verify signature (v1/v2/v3/v4) apksigner verify --verbose --print-certs app.apk # Quick metadata via aapt2 aapt2 dump badging app.apk | head -40
▲Signature v1 only → JAR signing; vulnerable to Janus attack (CVE-2017-13156)
▲debuggable=true in release → allow arbitrary code injection via adb
✓v3+ signature = key rotation support, stronger guarantee
2Full decompilation stack
apktooljadxbaksmali
▾

Run apktool for smali+resources, jadx for Java source reconstruction. Use both — they fail on different obfuscation patterns.

# Apktool: smali + decoded resources apktool d app.apk -o app_decoded --no-src -f # Jadx: Java source (with fallback options) jadx app.apk -d app_jadx --show-bad-code --deobf \ --deobf-min 3 --deobf-max 64 # baksmali for raw smali if jadx fails baksmali disassemble classes.dex -o smali_out/
▲Multiple classes.dex files → multi-dex; analyse all of them
▲JADX "bad code" markers indicate obfuscated flow — treat as high-risk
3File tree survey & entropy map
unzippython3binwalk
▾

High entropy blobs in assets/ or res/ are encrypted payloads, packed DEX, or embedded APKs. Map them before going deeper.

unzip -l app.apk | sort -k3 -rn | head -30 # Entropy scan (Python one-liner per file) python3 -c " import os, math def ent(f): d=open(f,'rb').read() if not d: return 0 freq=[d.count(bytes([i]))/len(d) for i in range(256)] return -sum(p*math.log2(p) for p in freq if p) for r,_,fs in os.walk('.'): for f in fs: p=os.path.join(r,f); e=ent(p) if e>7.2: print(f'{e:.2f} {p}') " # Binwalk for embedded archives / magic bytes binwalk -e app.apk
▲Entropy > 7.2 in .bin/.dat/.so → likely packed payload or encrypted DEX
▲assets/classes.dex not in META-INF → dynamic loading, evades basic scanning
Phase 2AndroidManifest Audit

The manifest is the attack surface declaration. Every exported component, permission, and intent filter is a potential entry point.

1Permission analysis
grepaapt2python3
▾

Map declared permissions against what the code actually uses. Unused dangerous permissions are a red flag — possible dormant capability.

# Extract all permissions grep -oP '(?<=android:name=")[^"]+' app_decoded/AndroidManifest.xml \ | grep "permission\." | sort -u # Dangerous permissions (runtime-requestable) DANGER="READ_CONTACTS|RECORD_AUDIO|ACCESS_FINE_LOCATION|\ READ_CALL_LOG|READ_SMS|CAMERA|READ_EXTERNAL_STORAGE|\ PROCESS_OUTGOING_CALLS|RECEIVE_SMS|BIND_ACCESSIBILITY_SERVICE" grep -E "$DANGER" app_decoded/AndroidManifest.xml
▲BIND_ACCESSIBILITY_SERVICE + RECORD_AUDIO = spyware pattern
▲RECEIVE_BOOT_COMPLETED + INTERNET = persistence + C2 capability
▲REQUEST_INSTALL_PACKAGES = dropper capability
2Exported component enumeration
grepdrozer
▾

Exported Activities, Services, Receivers, and Providers without permission checks are directly attackable by any installed app.

# Count exported components grep -c 'android:exported="true"' app_decoded/AndroidManifest.xml # Find exported without permission python3 - << 'EOF' import re with open('app_decoded/AndroidManifest.xml') as f: xml = f.read() comp = re.findall( r'<(activity|service|receiver|provider)[^>]+exported="true"[^>]*>', xml, re.S) no_perm = [c for c in comp if 'permission' not in c] print(f"{len(no_perm)} exported without permission check") for c in no_perm[:10]: print(' ', c[:120]) EOF
▲ContentProvider exported=true without readPermission → data theft via content:// URI
▲BroadcastReceiver exported for system actions — check for intent redirection
3Deep link & intent filter audit
greppython3
▾

Deep links without host/path validation are open redirectors and can deliver arbitrary payloads to the app.

# Extract all custom schemes / deep links grep -A5 'intent-filter' app_decoded/AndroidManifest.xml \ | grep -E 'scheme|host|path' # Find http/https schemes (potential deep link hijacking) grep -B10 'android:scheme="https"' app_decoded/AndroidManifest.xml \ | grep -E 'activity|host'

The bulk of analysis. Use grep patterns for speed, then targeted CFG and data-flow analysis for depth.

1Dangerous API grep pass
grep / ripgrepsemgrep
▾

A targeted grep for high-signal API patterns catches 70% of common vulns in under 5 minutes.

▾

3Obfuscation & packing detection
apkiddexdump
▾

Phase 4Strings, Secrets & Hardcoded Indicators

Strings in APKs are rarely encrypted. Extract them from all layers — Java source, smali, native .so — and run them through multiple detection passes.

1Multi-layer string extraction
stringsapktool
▾
# Strings from all .so files find app_decoded/lib -name "*.so" -exec strings {} \; > native_strings.txt # Strings from decoded resources (res/values/strings.xml) grep -h ">" app_decoded/res/values/strings.xml | \ sed 's/<[^>]*>//g' | sort -u > resource_strings.txt # All strings from smali (quoted literals) grep -roh '"[^"]\{8,\}"' app_decoded/smali*/ | \ sed 's/.*:"//' | tr -d '"' | sort -u > smali_strings.txt cat native_strings.txt smali_strings.txt resource_strings.txt \ | sort -u > all_strings.txt wc -l all_strings.txt

Phase 5Native Libraries & Resource Analysis

Native .so files are a blind spot for most APK scanners. They can bypass DEX-level controls and contain the most critical logic.

1Native library audit
Ghidrar2nm / readelf
▾

Check every .so for JNI exports, suspicious imports, and non-standard function names that suggest custom packer stubs.

# List all native libraries by arch find app_decoded/lib -name "*.so" | sort # JNI exports (callable from Java) for f in app_decoded/lib/arm64-v8a/*.so; do echo "=== $f ==="; nm -D "$f" 2>/dev/null | grep "Java_" done # Suspicious imports (network, file, crypto) readelf -d app_decoded/lib/arm64-v8a/libsuspect.so | grep NEEDED nm -D app_decoded/lib/arm64-v8a/libsuspect.so | \ grep -E "socket|connect|recv|fork|exec|dlopen|mmap" # Ghidra headless analysis (batch) $GHIDRA_HOME/support/analyzeHeadless /tmp/ghidra_proj myproj \ -import app_decoded/lib/arm64-v8a/*.so \ -postScript FindStrings.java \ -scriptPath ~/ghidra_scripts

Phase 7 — NovelWhat MobSF & VirusTotal Miss

MobSF runs pattern matching and known-bad heuristics. VirusTotal runs AV signatures. Neither does deep semantic reasoning about behavior intent, capability composition, or trust boundary violations at the smali level. This phase fills that gap.

1Capability composition fingerprinting
python3Claude API
▾

The gap MobSF has

MobSF flags individual permissions as "dangerous". It does not reason about permission combinations that together constitute a known malware archetype — even when no single permission is alarming. A stalkerware may request LOCATION (reasonable), CONTACTS (reasonable), and RECORD_AUDIO (reasonable) — each passes MobSF's heuristics. The combination doesn't.

MobSF gap

MobSF shows a flat list of "dangerous permissions". It does not compute archetype-level intent from combinations. This script does.

2Semantic intent classification with LLM
Claude APIpython3
▾

The gap VirusTotal has

VirusTotal passes APKs to 60+ AV engines. Every engine does signature matching — comparing file/code patterns to known-malicious databases. Novel malware, repacked samples, or newly written spyware with 0 prior detections scores 0/60. No engine reasons about what the code intends to do from first principles.

VirusTotal gap

60 AV engines, 0 on novel malware. An LLM reading the decompiled code reasons from semantics, not signatures. It can flag a novel banking trojan on first encounter because it understands what accessibility service abuse means.

3Trust boundary violation graph
python3networkxsmali grep
▾

The gap both tools share

Neither MobSF nor VirusTotal models data flows across trust boundaries at the smali level — i.e., which user-controlled inputs flow into exported components that other apps can reach, which flows reach native code, and which reach IPC mechanisms. This is where privilege escalation chains live.

Shared gap in both tools

This graph exposes privilege escalation chains: external apps → exported component → dangerous sink. This is the static equivalent of "confused deputy" attack mapping — not available in MobSF or VirusTotal.

4C2 infrastructure fingerprinting via ML
DGA detectorsklearnvirustotal API
▾

Why standard IoC lookup fails

VirusTotal looks up known-bad domains/IPs. DGA (Domain Generation Algorithm) domains are generated fresh — they haven't been seen before, so lookup returns clean. A trained classifier on domain n-grams can catch DGA domains at near-zero false positive rate even on first encounter.

VirusTotal gap

VT's domain reputation looks up known-bad lists. DGA domains are generated fresh per campaign. A DGA detector catches C2 infrastructure that hasn't been reported yet.

5Smali-level anti-analysis detection
greppython3
▾

Why this matters

MobSF runs in an automated pipeline and does not check whether the app detects it's being analyzed and changes behavior. Anti-analysis code is the clearest signal that malicious behavior is intentionally hidden — a benign app has no reason to detect debuggers, emulators, or hook frameworks.

MobSF gap

MobSF does not enumerate anti-analysis patterns comprehensively or flag their combination as a high-confidence maliciousness signal. Any app checking for Frida, emulator, and debugger is deliberately hiding behavior.

Summary: what this phase adds over existing tools

Capability composition fingerprinting catches archetype-level intent from permission combinations. LLM semantic classification reasons from decompiled code rather than comparing signatures — catches day-zero malware. Trust boundary graphs expose privilege escalation chains invisible to both tools. DGA detection catches fresh C2 infrastructure before it's reported to VirusTotal. Anti-analysis enumeration flags deliberate behavior hiding — a near-certain maliciousness signal.
